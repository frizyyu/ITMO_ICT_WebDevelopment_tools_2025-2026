Лабораторная работа 2

Потоки, процессы и асинхронность

В работе реализованы две части:

  1. Вычисление суммы чисел от 1 до 10 000 000 000 000.
  2. Параллельный парсинг веб-страниц с проектами и сохранение полученных данных в базу данных из ЛР1.

Цель работы

Изучить различия между потоками, процессами и асинхронностью в Python, а также проверить, какие подходы лучше подходят для CPU-bound и I/O-bound задач.

Задача 1. Вычисление суммы

Для первой задачи реализованы три отдельные программы:

  • threading_sum.py;
  • multiprocessing_sum.py;
  • asyncio_sum.py.

Каждая программа содержит функцию calculate_sum(...), разбивает диапазон на несколько чанков, измеряет время выполнения через time.perf_counter() и проверяет результат формулой:

n * (n + 1) // 2

Сумма внутри каждого чанка вычисляется по формуле арифметической прогрессии:

count = end - start + 1
chunk_sum = (start + end) * count // 2

Такой способ выбран потому, что прямой цикл по 10 000 000 000 000 числам выполнялся бы слишком долго. Формула не меняет математический результат: она дает ту же сумму чисел на отрезке, но вычисляет ее за константное время.

Threading

В threading_sum.py диапазон делится на чанки, для каждого чанка создается поток. Потоки записывают частичные результаты, затем основной поток объединяет их в итоговую сумму.

Потоки удобны для I/O-bound задач, но для CPU-bound вычислений в CPython ограничены GIL.

Multiprocessing

В multiprocessing_sum.py для чанков создаются отдельные процессы. Результаты возвращаются через multiprocessing.Queue.

Процессы лучше подходят для CPU-bound задач, потому что у каждого процесса отдельный интерпретатор Python и отдельный GIL. В данной работе вычисление по формуле очень быстрое, поэтому в замерах заметны накладные расходы на запуск процессов.

Asyncio

В asyncio_sum.py чанки запускаются как асинхронные задачи через asyncio.gather(...).

Этот вариант показывает асинхронную оркестрацию, но не дает настоящего CPU-параллелизма. asyncio полезен прежде всего там, где программа ожидает ввод-вывод.

Результаты задачи 1

Результаты получены после запуска run_all_benchmarks.py.

Approach Chunks Verified Time, seconds Result
threading 8 yes 0.001520 50 000 000 000 005 000 000 000 000
multiprocessing 8 yes 0.832878 50 000 000 000 005 000 000 000 000
asyncio 8 yes 0.000126 50 000 000 000 005 000 000 000 000

Все три подхода получили одинаковый результат и прошли проверку.

Задача 2. Парсинг и заполнение БД ЛР1

Во второй задаче реализованы три программы:

  • threading_parse.py;
  • multiprocessing_parse.py;
  • asyncio_parse.py.

Каждая программа содержит функцию parse_and_save(url).

Парсер обрабатывает публичные страницы PeoplePerHour с карточками проектов. Эти страницы выбраны потому, что они отдают данные в обычном HTML и не требуют авторизации.

Используемые источники:

  • https://www.peopleperhour.com/freelance-jobs/technology-programming
  • https://www.peopleperhour.com/freelance-jobs/technology-programming/programming-coding
  • https://www.peopleperhour.com/freelance-jobs/technology-programming/website-development
  • https://www.peopleperhour.com/freelance-jobs/technology-programming/e-commerce-cms-development
  • https://www.peopleperhour.com/freelance-jobs/artificial-intelligence/artificial-intelligence-agent-development
  • https://www.peopleperhour.com/freelance-jobs/artificial-intelligence/artificial-intelligence-website-development

С каждой страницы извлекаются карточки проектов. Для каждой карточки берутся:

  • название проекта;
  • ссылка на проект;
  • текст описания;
  • бюджет, если он указан в карточке;
  • категория источника.

Как данные сохраняются в БД ЛР1

В ЛР1 реализован менеджер задач. Поэтому данные, полученные парсером, сохраняются в существующие таблицы ЛР1:

  • users;
  • categories;
  • tasks;
  • task_categories.

Создается технический пользователь:

parser_bot

Каждая карточка проекта сохраняется как задача:

project title       -> tasks.title
project description -> tasks.description
project URL         -> tasks.description
budget              -> tasks.description
parser method       -> tasks.description
source category     -> categories.name

Связь задачи с категорией сохраняется через таблицу task_categories.

Для защиты от дублей в tasks.description сохраняется служебный маркер:

Source ID: lr2:<parser_method>:<project_url>

Если при повторном запуске задача с таким Source ID уже существует, она обновляется. Если появляется новый проект с новым URL, создается новая задача.

Реализация задачи 2 по подходам

Threading

threading_parse.py использует threading и requests.

URL делятся на группы. Каждая группа обрабатывается отдельным потоком. Такой подход подходит для сетевых задач, потому что во время ожидания HTTP-ответа один поток не блокирует выполнение остальных.

Multiprocessing

multiprocessing_parse.py использует процессы и requests.

Подход корректно выполняет задачу, но для сетевого парсинга имеет дополнительные накладные расходы: создание процессов, отдельная память и передача результатов через Queue.

Asyncio

asyncio_parse.py использует aiohttp и asyncio.gather(...).

HTTP-запросы выполняются конкурентно. Сохранение в БД выполняется синхронным SQLAlchemy-кодом, поэтому оно вынесено через asyncio.to_thread(...).

Для I/O-bound задачи этот подход оказался самым быстрым.

Результаты задачи 2

Результаты получены после запуска run_all_benchmarks.py.

Approach URLs Saved tasks Errors Time, seconds
threading 6 30 0 2.227198
multiprocessing 6 30 0 2.522838
asyncio 6 30 0 1.019772

Каждый подход обработал 6 страниц и выполнил по 30 операций сохранения задач.

После очистки БД и повторной проверки в таблице tasks осталось 75 задач ЛР2. Это число может отличаться от количества операций сохранения, потому что PeoplePerHour является динамическим источником:

  • уже известные карточки обновляются по Source ID;
  • новые карточки добавляются как новые задачи;
  • часть карточек может повторяться на разных страницах.

Различия threading, multiprocessing и asyncio

threading запускает несколько потоков внутри одного процесса. Потоки разделяют память. Они полезны для I/O-bound задач, но при CPU-bound вычислениях ограничены GIL.

multiprocessing запускает отдельные процессы. У каждого процесса свой интерпретатор Python и свой GIL, поэтому этот подход лучше подходит для тяжелых CPU-bound задач.

asyncio использует event loop и позволяет эффективно выполнять большое количество операций ожидания: HTTP-запросы, сетевой ввод-вывод, обращения к внешним API. Asyncio не заменяет процессы для тяжелых вычислений.

GIL, или Global Interpreter Lock, не позволяет нескольким потокам одновременно выполнять Python-байткод внутри одного процесса. Поэтому:

  • для CPU-bound задач обычно лучше использовать multiprocessing;
  • для I/O-bound задач подходят threading и asyncio;
  • asyncio особенно полезен при большом количестве сетевых запросов.

Вывод

В работе реализованы три подхода к конкурентному выполнению задач в Python.

Для вычислительной задачи все подходы корректно посчитали сумму и прошли проверку результата.

Для задачи парсинга самым быстрым оказался asyncio, так как парсинг веб-страниц является I/O-bound задачей и основное время уходит на ожидание сетевых ответов.

База данных из ЛР1 была заполнена данными, реально полученными при парсинге: карточки проектов PeoplePerHour преобразованы в задачи приложения управления задачами.
