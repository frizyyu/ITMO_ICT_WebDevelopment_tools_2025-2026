Лабораторная работа 3¶

Информация¶

Тема: упаковка FastAPI приложения в Docker, работа с источниками данных и очереди.

Студент: Васильев Артём
Группа: K3340

Цель работы: разработать и упаковать в Docker Compose систему на базе FastAPI, PostgreSQL, отдельного сервиса парсинга, Redis, Celery worker и Celery Beat. Основное API должно уметь вызывать parser-service синхронно по HTTP и асинхронно через очередь задач.

Краткое описание проекта¶

Из ЛР1 была взята основа FastAPI-приложения:

  • структура app/, core/, models/, repositories/, schemas/, services/;
  • SQLAlchemy-модели;
  • Alembic-миграции;
  • PostgreSQL-конфигурация;
  • существующие endpoint-ы API, включая GET /health и GET /api/v1/health.

Из ЛР2 была взята логика парсера:

  • parser/common_parse.py;
  • parser/asyncio_parse.py;
  • функции разбора страниц и сохранения результатов в БД;
  • список URL по умолчанию DEFAULT_URLS.

В ЛР3 реализовано:

  • Dockerfile для основного API;
  • Dockerfile для отдельного parser-service;
  • docker-compose.yml для запуска всех сервисов;
  • PostgreSQL в контейнере;
  • Redis в контейнере;
  • Celery worker;
  • Celery Beat;
  • синхронный HTTP-вызов parser-service из API;
  • асинхронный запуск парсера через Celery;
  • endpoint проверки статуса фоновой задачи;
  • периодическая задача парсинга.

Архитектура¶

В проекте используются следующие сервисы:

  • api — основное FastAPI-приложение, доступно на localhost:8000;
  • parser-service — отдельный FastAPI-сервис для запуска парсера, доступен на localhost:8001;
  • postgres — база данных PostgreSQL, внутри Docker доступна как postgres:5432;
  • redis — брокер сообщений и result backend для Celery, внутри Docker доступен как redis:6379;
  • celery-worker — фоновый обработчик задач парсинга;
  • celery-beat — планировщик периодических задач.

Текстовая схема:

Client
  |
  v
FastAPI API
  |
  +--> parser-service через HTTP
  |
  +--> Redis queue
           |
           v
     Celery worker
           |
           v
     parser-service
           |
           v
     PostgreSQL

Описание Docker¶

Dockerfile основного API¶

Файл Dockerfile в корне Lr3:

  • использует образ python:3.11-slim;
  • задает рабочую директорию /app;
  • устанавливает зависимости из requirements.txt;
  • копирует код приложения;
  • запускает API командой:
uvicorn app.main:app --host 0.0.0.0 --port 8000

Dockerfile parser-service¶

Файл parser_service/Dockerfile:

  • использует образ python:3.11-slim;
  • устанавливает зависимости из requirements.txt;
  • копирует папки parser/ и parser_service/;
  • запускает parser-service командой:
uvicorn parser_service.main:app --host 0.0.0.0 --port 8001

docker-compose.yml¶

Файл docker-compose.yml описывает сервисы:

  • postgres;
  • redis;
  • api;
  • parser-service;
  • celery-worker;
  • celery-beat.

Используемые порты:

  • 8000:8000 — основное API;
  • 8001:8001 — parser-service;
  • 5432:5432 — PostgreSQL;
  • 6379:6379 — Redis.

Для PostgreSQL настроен healthcheck через pg_isready. Для Redis настроен healthcheck через redis-cli ping.

Описание базы данных¶

В проекте используется PostgreSQL. Подключение внутри Docker выполняется через:

DATABASE_URL=postgresql+psycopg://postgres:postgres@postgres:5432/time_manager

Для миграций используется Alembic. Миграции находятся в папке alembic/.

Основные таблицы после применения миграций:

  • users;
  • categories;
  • tasks;
  • task_categories;
  • time_logs;
  • daily_plans;
  • daily_plan_tasks;
  • notifications;
  • alembic_version.

Парсер сохраняет данные в существующие таблицы приложения:

  • создает или находит технического пользователя parser_bot;
  • создает или находит категорию;
  • создает или обновляет записи в tasks;
  • связывает задачи и категории через task_categories.

Описание parser-service¶

Parser-service находится в папке parser_service/.

Реализованные endpoint-ы:

GET /health
POST /parse

GET /health¶

Проверяет, что parser-service запущен.

Ожидаемый ответ:

{"status":"ok","service":"parser-service"}

POST /parse¶

Принимает JSON:

{
  "urls": ["https://www.peopleperhour.com/freelance-jobs/technology-programming"],
  "workers": 4
}

Возвращает JSON-результат парсинга:

{
  "method": "asyncio",
  "started_at": 123.45,
  "urls": 1,
  "saved": 5,
  "extracted": 5,
  "ok": 1,
  "errors": 0,
  "elapsed_seconds": 1.23,
  "items": []
}

Фактические значения saved, extracted, elapsed_seconds и items зависят от доступности внешнего сайта и текущей HTML-структуры страницы.

Parser-service использует код парсера из ЛР2, перенесенный в папку parser/.

Синхронный вызов парсера¶

Основное API содержит endpoint:

POST /api/v1/parser/parse-sync

Схема работы:

Client -> FastAPI API -> HTTP POST parser-service /parse -> PostgreSQL -> FastAPI API -> Client

Пример PowerShell-команды:

curl.exe -X POST http://localhost:8000/api/v1/parser/parse-sync `
  -H "Content-Type: application/json" `
  -d '{\\\"urls\\\":[\\\"https://www.peopleperhour.com/freelance-jobs/technology-programming\\\"],\\\"workers\\\":4}'

Ожидаемый результат: API вернет JSON-ответ parser-service. При успешной работе поле ok равно 1, а errors равно 0.

Асинхронный вызов через очередь¶

Для асинхронного запуска используются Redis и Celery.

Redis используется как:

  • broker для передачи задач;
  • result backend для хранения результата выполнения.

Celery worker выполняет задачу:

worker.tasks.parse_urls_task

Endpoint постановки задачи в очередь:

POST /api/v1/parser/parse-async

Пример PowerShell-команды:

$response = curl.exe -s -X POST http://localhost:8000/api/v1/parser/parse-async `
  -H "Content-Type: application/json" `
  -d '{\\\"urls\\\":[\\\"https://www.peopleperhour.com/freelance-jobs/technology-programming\\\"],\\\"workers\\\":4}' | ConvertFrom-Json

$taskId = $response.task_id
$response

Ожидаемый ответ:

{
  "task_id": "celery-task-id",
  "status": "queued"
}

Endpoint проверки статуса:

GET /api/v1/parser/tasks/{task_id}

Пример:

curl.exe http://localhost:8000/api/v1/parser/tasks/$taskId

Ожидаемый результат после выполнения:

{
  "task_id": "celery-task-id",
  "status": "SUCCESS",
  "result": {
    "method": "asyncio",
    "urls": 1,
    "saved": 5,
    "extracted": 5,
    "ok": 1,
    "errors": 0
  },
  "error": null
}

Periodic tasks¶

Для периодического запуска задач используется Celery Beat.

В worker/celery_app.py настроена периодическая задача:

worker.tasks.parse_default_urls_task

Периодичность:

каждые 30 минут

Часовой пояс:

UTC

Задача использует DEFAULT_URLS из parser/common_parse.py и вызывает parser-service через Celery worker. Это нужно для автоматического регулярного обновления данных без ручного HTTP-запроса.

Проверка PostgreSQL¶

Список таблиц:

docker compose exec postgres psql -U postgres -d time_manager -c "\dt"

Последние записи в таблицах:

docker compose exec postgres psql -U postgres -d time_manager -c "select id, username, email from users order by id desc limit 10;"
docker compose exec postgres psql -U postgres -d time_manager -c "select id, name, owner_id from categories order by id desc limit 10;"
docker compose exec postgres psql -U postgres -d time_manager -c "select id, title, owner_id from tasks order by id desc limit 10;"
docker compose exec postgres psql -U postgres -d time_manager -c "select id, task_id, category_id, label from task_categories order by id desc limit 10;"

Вывод¶

В ходе лабораторной работы были изучены и применены:

  • упаковка FastAPI-приложения в Docker;
  • запуск нескольких сервисов через Docker Compose;
  • работа PostgreSQL в контейнере;
  • вынесение parser-service в отдельный микросервис;
  • HTTP-взаимодействие между сервисами;
  • Redis как брокер сообщений и хранилище результатов;
  • Celery для фоновых задач;
  • endpoint-ы для асинхронного запуска и проверки статуса;
  • Celery Beat для periodic tasks.

В результате получилась система, где основной API может запускать парсер напрямую или через очередь, а результаты сохраняются в PostgreSQL.
